# Проект "Система мониторинга загрязнения воздуха"
## Состав команды - Курило Наталья, группа 1/233
## Карточка проекта (итерация 1):
### Назначение:
#### Создание сети датчиков для непрерывного измерения уровня загрязнения воздуха в различных районах города и предоставления данных для анализа и принятия решений по улучшению экологической ситуации.
### Мотивация (почему):
- Рост уровня загрязнения воздуха в городах приводит к ухудшению здоровья населения и окружающей среды;
- Существующие системы мониторинга могут быть недостаточно подробными или доступными для анализа ситуации в реальном времени;
- Разработка собственной системы мониторинга позволит получить точные и актуальные данные о состоянии воздуха в городе, что поможет принимать обоснованные решения по снижению загрязнения.
### Чего хотите добиться:
- Создание сети датчиков, охватывающей различные районы города, для получения полных данных о состоянии воздуха;
- Разработка программного обеспечения для сбора, обработки и анализа данных с датчиков;
- Предоставление данных о загрязнении воздуха в удобном формате для использования органами власти, экологическими организациями и населением;
- Повышение осведомлённости о проблеме загрязнения воздуха и стимулирование действий по его снижению.
### Общие свойства:
|Функции |Качественные характеристики |
|-------|---------------------------|
|1) Непрерывное измерение уровня загрязнения воздуха (концентрации вредных веществ) в различных точках города.|1) Высокая точность и надёжность измерений.|
|2) Передача данных с датчиков в централизованную систему для обработки и анализа.|2) Быстрое обновление данных в реальном времени.|
|3) Визуализация данных на карте города с указанием уровня загрязнения в каждом районе.|3) Простота использования и настройки системы.|
|4) Возможность сравнения данных за разные периоды времени для анализа динамики загрязнения.|4) Возможность расширения сети датчиков для покрытия новых районов города.|
|5) Интеграция с другими системами мониторинга окружающей среды для более полного анализа ситуации.| |

## Итерация 4
### Общий системный контекст
**Система мониторинга загрязнения воздуха** взаимодействует с различными внешними системами и пользователями. Вот некоторые из них:

**Пользователи**:
- **Экологические службы и агентства**: получают данные о качестве воздуха для принятия решений и разработки политики.
- **Научные исследователи**: используют данные для изучения влияния загрязнения на здоровье и окружающую среду.
- **Городские власти**: применяют информацию для планирования городских проектов и улучшения качества жизни.
- **Граждане**: интересуются качеством воздуха в своём районе для принятия личных решений.

**Внешние системы**:
- **Датчики качества воздуха**: предоставляют данные о концентрации загрязняющих веществ.
- **Системы метеорологических данных**: предоставляют информацию о погодных условиях, которые могут влиять на распространение загрязняющих веществ.
- **Географические информационные системы (ГИС)**: предоставляют данные о местоположении датчиков и других объектов.

### Интерфейс взаимодействия
**Структура и формат обмениваемых данных**:
1) **Данные о качестве воздуха**: концентрация загрязняющих веществ (например, PM2.5, PM10, NO2, SO2), время и место измерения.
2) **Метеорологические данные**: температура, влажность, скорость и направление ветра.
3) **Географические данные**: координаты датчиков, границы районов и т. д.

**Способ обмена**:
1) **Синхронный обмен**: используется для получения данных в реальном времени, например, для отображения текущей концентрации загрязняющих веществ на карте.
2) **Асинхронный обмен**: используется для передачи больших объёмов данных, например, для анализа исторических данных о качестве воздуха.

**Объёмы обмениваемых данных**:
- Зависит от количества датчиков, частоты измерений и объёма метеорологических и географических данных.
- Необходимо учитывать пропускную способность каналов связи и предполагаемую нагрузку на систему.

**Количество запросов и интенсивность**:
- Запросы на получение данных о качестве воздуха могут поступать от различных пользователей и внешних систем.
- Интенсивность запросов зависит от частоты измерений и потребности пользователей в данных.

**Распределение по времени**:
- Запросы могут поступать в любое время суток, но наиболее интенсивные периоды могут быть связаны с пиковыми нагрузками, например, в часы пик или во время неблагоприятных метеорологических условий.

### Внутренняя структура системы
**Контейнеры и компоненты**:
- **Модуль сбора данных**: собирает данные с датчиков качества воздуха, метеорологических станций и других источников.
- **Модуль обработки данных**: обрабатывает собранные данные, применяя алгоритмы фильтрации, нормализации и агрегации.
- **Модуль хранения данных**: хранит обработанные данные в базе данных.
- **Модуль анализа данных**: анализирует данные, выявляя тенденции и аномалии.
- **Модуль визуализации данных**: отображает данные на карте, графиках и других визуализациях.
- **Модуль API**: предоставляет интерфейс для взаимодействия с внешними системами и пользователями.

![image](https://github.com/user-attachments/assets/e87de6d1-3b58-4250-9e23-523122d69052)

**Связь между модулями**:
- **Модуль сбора данных** собирает информацию с датчиков и других источников и передаёт её для дальнейшей обработки — поэтому стрелка идёт к **модулю обработки данных**.
- **Модуль обработки данных** получает сырые данные, применяет к ним алгоритмы фильтрации и нормализации, после чего передаёт обработанную информацию для хранения — отсюда стрелка к **модулю хранения данных**.
- **Модуль анализа данных** работает с данными, которые уже сохранены в базе данных, поэтому стрелка идёт от **модуля хранения данных** к **модулю анализа данных**.
- **Модуль визуализации данных** отображает результаты анализа, полученные из модуля анализа данных, — отсюда стрелка от **модуля анализа данных** к **модулю визуализации данных**.
- **Модуль API** предоставляет доступ к данным и функциям системы внешним системам и пользователям, но не участвует напрямую в потоке данных между другими модулями, поэтому он представлен отдельно.
